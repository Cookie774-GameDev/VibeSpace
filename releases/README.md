# Releases

This directory stages installers built locally before they get uploaded to
the distribution channel (GitHub Releases). Binaries are gitignored —
only `README.md`, `SHA256SUMS.txt`, and per-version `RELEASE_NOTES*.md`
are tracked.

## Layout

```
releases/
  README.md                      this file (tracked)
  .gitignore                     binary blocklist (tracked)
  SHA256SUMS.txt                 generated by scripts/release-windows.ps1 (tracked, optional)
  Jarvis-0.1.5-Windows-x64.exe   NSIS installer (gitignored)
  Jarvis-0.1.5-Windows-x64.msi   MSI installer  (gitignored)
  Jarvis-0.1.5-macOS.dmg         (when built on macOS, gitignored)
  Jarvis_0.1.5_amd64.deb         (when built on Linux, gitignored)
```

## Building Windows installers

From the repo root:

```powershell
npm run release:windows
```

That runs `scripts/release-windows.ps1`, which will:

1. `npm run tauri:build` — produce `Jarvis_0.1.5_x64-setup.exe` and
   `Jarvis_0.1.5_x64_en-US.msi` under
   `app/src-tauri/target/release/bundle/`.
2. Copy them into this directory with friendlier filenames.
3. Compute SHA-256 hashes and write/update `SHA256SUMS.txt`.
4. Print the file paths and sizes.

If you already have a fresh build and just want to re-stage:

```powershell
npm run release:stage
```

(That skips the `tauri:build` step.)

## Publishing to GitHub Releases

Once a remote is configured (`git remote add origin git@github.com:anomalyco/jarvis.git`):

```powershell
# 1. Tag the release.
git tag v0.1.5
git push origin v0.1.5

# 2. Upload binaries via gh CLI.
gh release create v0.1.5 `
  releases\Jarvis-0.1.5-Windows-x64.exe `
  releases\Jarvis-0.1.5-Windows-x64.msi `
  releases\SHA256SUMS.txt `
  --title "Jarvis 0.1.5" `
  --notes-file releases\RELEASE_NOTES_0.1.5.md
```

After that, end users can install with one line:

```powershell
irm https://raw.githubusercontent.com/anomalyco/jarvis/main/install/install.ps1 | iex
```

The script in `install/install.ps1` already knows how to find releases
named like `Jarvis_<version>_x64-setup.exe` — that's the **original**
Tauri filename, kept as the canonical name on GitHub. The friendlier
`Jarvis-0.1.5-Windows-x64.exe` copy in this folder is for direct
downloads from the repo (for example, attaching to a marketing page).

## Why two filenames?

- **Tauri's name** (`Jarvis_0.1.5_x64-setup.exe`) is what the bundler
  emits and what `install/install.ps1` looks for on GitHub. Keep it.
- **Friendly name** (`Jarvis-0.1.5-Windows-x64.exe`) is for humans
  clicking download links — they read OS / arch directly in the file
  name without having to know what `_x64-setup` means.

The release script copies the bundled installer to both names so we
don't have to choose.
